# 物资管理数据库设计

## 1. 设计说明

本设计依据 [物资管理需求说明](file:///d:/workspace/workspace-zkhf/zkhf-epmis-ai-en/docs/%E7%89%A9%E8%B5%84%E7%AE%A1%E7%90%86%E9%9C%80%E6%B1%82%E8%AF%B4%E6%98%8E.md) 进行拆解，采用“基础资料 + 库存台账 + 业务单据 + 明细 + 流水”模型。

设计原则如下：

1. 复用现有企业维度字段 `ent_code` 进行企业权限控制。
2. 复用现有附件表 `t_data_annex` 存储附件，不在业务表中重复建附件字段。
3. 复用现有审批实例表 `t_bas_approval_instance` 存储审批流转数据，业务表仅保留业务状态与审核结果快照。
4. 单据表与明细表分离，满足一单多物资场景。
5. 库存采用汇总表 + 流水表模式，便于快速查询与审计追溯。
6. 表主键统一建议使用 `varchar(26)`，便于与现有 ULID 风格保持一致。

## 2. 表清单

### 2.1 基础资料表

1. `t_material_info`：物资基础信息表
2. `t_material_warehouse`：仓库信息表

### 2.2 库存台账表

1. `t_material_stock`：物资库存汇总表
2. `t_material_stock_flow`：物资库存流水表

### 2.3 业务单据表

1. `t_material_in_order`：物资入库单
2. `t_material_in_order_item`：物资入库单明细
3. `t_material_apply_order`：物资申请单
4. `t_material_apply_order_item`：物资申请单明细
5. `t_material_out_order`：物资出库单
6. `t_material_out_order_item`：物资出库单明细
7. `t_material_return_order`：物资归还单
8. `t_material_return_order_item`：物资归还单明细

## 3. 关系说明

1. 一个企业可拥有多个仓库、多个物资、多个业务单据。
2. 一个仓库可存放多个物资，一个物资也可出现在多个仓库中，通过 `t_material_stock` 维护“企业 + 仓库 + 物资”唯一库存。
3. 入库单、申请单、出库单、归还单均采用主表 + 明细表结构。
4. 出库单可关联申请单，建议通过 `t_material_out_order.apply_id` 建立关联。
5. 归还单可关联出库单，建议通过 `t_material_return_order.out_id` 建立关联。
6. 库存流水表通过 `biz_type + biz_id + biz_item_id` 关联各业务单据，记录库存变化过程。

## 4. 状态与编码建议

### 4.1 通用状态

1. `status` 为业务状态字段。
2. 物资相关表不使用 `del_flag`，未被引用的数据可物理删除（删除前需做引用校验）。

### 4.2 物资与仓库状态

1. `0`：正常/启用
2. `1`：停用

### 4.3 单据状态

1. `1`：草稿
2. `2`：待审核
3. `3`：已审核
4. `4`：已完成
5. `0`：已作废

### 4.4 申请单审核状态

1. `1`：草稿
2. `2`：待审核
3. `3`：审核通过
4. `4`：审核驳回
5. `5`：已取消
6. `6`：已保存

### 4.5 申请单出库状态

1. `0`：未出库
2. `1`：部分出库
3. `2`：已出库

### 4.6 归还状态

1. `0`：未归还
2. `1`：部分归还
3. `2`：已归还

### 4.7 库存状态

1. `1`：正常
2. `2`：低库存
3. `3`：无库存

### 4.8 库存流水业务类型

1. `IN`：入库
2. `OUT`：出库
3. `RETURN`：归还入库
4. `ADJUST`：库存调整
5. `FREEZE`：库存冻结
6. `UNFREEZE`：库存解冻

## 5. 公共能力复用

### 5.1 附件表复用

附件统一复用现有 `t_data_annex`，建议使用以下 `source_type`：

1. `material_info`
2. `material_in_order`
3. `material_apply_order`
4. `material_out_order`
5. `material_return_order`

### 5.2 审批表复用

审批统一复用现有 `t_bas_approval_instance`，建议使用以下 `business_type`：

1. `material_apply`
2. `material_in`
3. `material_out`
4. `material_return`

审批终态通过 MQTT 推送给业务层，主题 `approval_result`，消息体以 `business_type + business_key` 作为业务关联键。

## 6. 关键表字段说明

### 6.1 `t_material_info`

存储物资主数据，作为入库、申请、出库、归还的基础来源。

关键约束：

1. `ent_code + material_code` 唯一。
2. 停用物资不允许继续参与新业务。

### 6.2 `t_material_warehouse`

存储仓库基础信息。

关键约束：

1. `ent_code + warehouse_code` 唯一。
2. 停用仓库不允许继续创建新单据。

### 6.3 `t_material_stock`

存储库存汇总数据。

关键约束：

1. `ent_code + warehouse_id + material_id` 唯一。
2. `current_qty`、`available_qty`、`frozen_qty` 均为非负数。
3. `available_qty = current_qty - frozen_qty`。

### 6.4 `t_material_stock_flow`

存储每一次库存变动流水。

关键约束：

1. 任何库存变化都必须落流水。
2. 通过 `biz_type + biz_id + biz_item_id` 可追溯来源单据。

### 6.5 单据主表与明细表

主表用于保存单据头信息、状态、审核信息和汇总字段；明细表用于保存物资快照、数量、金额和备注。

设计说明：

1. 明细表冗余保存 `material_code`、`material_name`、`brand`、`model_spec`、`category_name`、`unit` 等快照，避免主数据变更影响历史单据展示。
2. 出库、归还明细表增加累计出库/归还数量字段，便于判断部分出库和部分归还状态。
3. 归还明细表增加 `stock_in_qty` 字段，支持损坏、报废后仅部分回补库存。
