# 前后端联调指南

## 1. 适用范围

本文档用于前后端、本地测试、接口联调与基础验收，覆盖登录认证、验证码、Token 传递、业务接口调用和 MQTT 数据链路验证。

## 2. 联调前准备

### 2.1 服务准备

建议至少启动以下服务：

1. `epmis-auth-service`
2. `epmis-platform-service`
3. `epmis-process-service`

若需联调设备接入或 MQTT 链路，再启动：

4. `epmis-protocol-service`
5. MQTT Broker

### 2.2 配置检查

联调前建议确认：

- MySQL、Redis 可访问
- 三个业务服务的 Feign 地址配置正确
- 三个业务服务的数据库名已统一或已明确分库
- 若开启验证码，Redis 可正常写入验证码缓存
- 若联调 MQTT，`mqtt.enabled`、`mqtt.broker-url` 等配置已生效

### 2.3 典型端口

- 认证服务：`9531`
- 数据处理服务：`9532`
- 平台服务：`9533`
- 协议服务：`9534`

## 3. 登录联调流程

### 3.1 获取验证码

请求：

```http
GET http://localhost:9531/captchaImage
```

典型返回：

```json
{
  "code": 200,
  "msg": "操作成功",
  "captchaEnabled": true,
  "uuid": "xxxxxxxx",
  "img": "Base64图片内容"
}
```

说明：

- `captchaEnabled=true` 时，登录必须携带 `code` 与 `uuid`
- `img` 为 Base64 图片，可在前端直接展示
- 如果 `captchaEnabled=false`，可忽略验证码字段

### 3.2 登录获取 Token

请求：

```http
POST http://localhost:9531/login
Content-Type: application/json

{
  "username": "admin",
  "password": "******",
  "code": "1234",
  "uuid": "xxxxxxxx"
}
```

登录参数字段说明：

- `username`：用户名
- `password`：密码
- `code`：验证码
- `uuid`：验证码唯一标识

典型返回：

```json
{
  "code": 200,
  "msg": "操作成功",
  "token": "eyJhbGciOiJIUzI1NiJ9..."
}
```

### 3.3 获取当前用户信息

请求头：

```http
Authorization: Bearer <token>
```

请求：

```http
GET http://localhost:9531/getInfo
```

用于验证：

- Token 是否有效
- 当前用户角色、权限是否正常返回

### 3.4 获取路由

请求：

```http
GET http://localhost:9531/getRouters
```

用于验证：

- 菜单数据是否完整
- 当前用户权限菜单是否正常

## 4. 前端对接要点

### 4.1 认证头传递

前端在登录成功后，需要将 token 放到请求头：

```http
Authorization: Bearer <token>
```

### 4.2 常见状态码处理

- `200`：调用成功
- `401`：未登录、Token 无效或已过期
- `403`：无权限访问
- `500`：服务端异常或业务异常

### 4.3 联调建议

- 登录后第一时间调用 `/getInfo` 和 `/getRouters`
- 若页面空白，优先检查菜单和权限接口是否正常
- 若业务接口 401，优先检查 token 是否写入请求头

## 5. 典型接口联调

### 5.1 平台字典接口

请求：

```http
POST http://localhost:9533/platform/dict/type/list
Content-Type: application/json
Authorization: Bearer <token>

{
  "dictName": "",
  "dictType": "",
  "status": ""
}
```

适用于验证：

- Token 鉴权
- 平台服务是否已正常连库
- 字典初始化数据是否已导入

### 5.2 应急预案接口

请求：

```http
POST http://localhost:9533/platform/emergency/plan/list
Content-Type: application/json
Authorization: Bearer <token>

{
  "entCode": "",
  "planId": "",
  "planName": "",
  "version": "",
  "riskUnit": "",
  "planType": null
}
```

适用于验证：

- 平台应急模块是否可用
- `t_emergency_plan` 是否已建表

### 5.3 在线监测接口

请求：

```http
POST http://localhost:9532/process/onlineMonitoring/output/list
Content-Type: application/json
Authorization: Bearer <token>
```

适用于验证：

- 数据处理服务是否可用
- 监测数据相关表是否已初始化

## 6. Windows `curl` 联调示例

### 6.1 获取验证码

```bash
curl.exe -s "http://localhost:9531/captchaImage"
```

### 6.2 登录

```bash
curl.exe -s -X POST "http://localhost:9531/login" ^
  -H "Content-Type: application/json" ^
  -d "{\"username\":\"admin\",\"password\":\"******\",\"code\":\"1234\",\"uuid\":\"你的uuid\"}"
```

### 6.3 获取用户信息

```bash
curl.exe -s "http://localhost:9531/getInfo" ^
  -H "Authorization: Bearer 你的token"
```

### 6.4 调用平台接口

```bash
curl.exe -s -X POST "http://localhost:9533/platform/emergency/plan/list" ^
  -H "Content-Type: application/json" ^
  -H "Authorization: Bearer 你的token" ^
  -d "{\"entCode\":\"\",\"planId\":\"\",\"planName\":\"\"}"
```

## 8. MQTT 联调验证

### 8.1 前提条件

- MQTT Broker 已启动
- `epmis-process-service` 已开启 `mqtt.enabled=true`
- `epmis-process-service` 与 `epmis-protocol-service` 指向同一 MQTT Broker

### 8.2 核心 topic

- `out_put_data_real`
- `out_put_data_minute`
- `out_put_data_hour`
- `out_put_data_day`
- `device_data_plc`

### 8.3 推荐验证顺序

1. 启动 Broker
2. 启动 `epmis-process-service`
3. 启动 `epmis-protocol-service`
4. 接入 HJ212 或 Modbus 设备
5. 查看协议服务日志是否出现 MQTT 发送记录
6. 查看数据处理服务日志是否出现 MQTT 接收记录
7. 检查数据库是否生成监测数据或 PLC 数据

### 8.4 验证重点

- Broker 是否出现两个客户端连接
- 发布与订阅的 `client-id` 是否冲突
- topic 是否与系统约定一致
- 数据处理服务是否出现按 topic 路由日志

## 9. 常见联调问题

### 9.1 验证码总是错误

排查方向：

- 是否先调用了 `/captchaImage`
- 登录时 `uuid` 是否与验证码图片返回的一致
- Redis 是否可用
- `sys.account.captchaEnabled` 是否开启

### 9.2 登录成功但业务接口 401

排查方向：

- 是否带上了 `Authorization: Bearer <token>`
- token 是否来自当前环境
- 认证服务和业务服务的 token 配置是否一致

### 9.3 页面没有菜单

排查方向：

- `/getRouters` 是否正常返回
- `sys_menu` 是否已初始化
- 当前用户是否有角色和菜单权限

### 9.4 业务接口 500

排查方向：

- 表是否存在
- 字典或基础数据是否已导入
- Feign 依赖服务是否启动
- 日志中是否有 SQL 或 NPE 异常

### 9.5 MQTT 没有数据

排查方向：

- `mqtt.enabled` 是否打开
- Broker 地址、用户名、密码是否正确
- 协议服务是否实际收到了设备报文
- MQTT topic 是否匹配

## 10. 联调建议

- 先完成登录、菜单、字典三步验证，再进入复杂业务
- 先用平台应急模块进行 CRUD 联调，再做设备数据联调
- 联调期间保留固定测试账号、企业编码和测试数据
- 对导入导出能力优先使用系统模板，不手写 Excel
