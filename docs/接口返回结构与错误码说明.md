# 接口返回结构与错误码说明

## 1. 说明

本项目大多数接口使用统一 JSON 返回结构，但仍有少量接口直接返回对象、列表或文件流。前后端联调时，建议优先按本文档区分处理。

## 2. 统一返回结构

项目主要使用两类响应：

- `AjaxResult`：普通接口、详情接口、保存接口、删除接口
- `TableDataInfo`：分页列表接口

## 3. `AjaxResult` 结构

### 3.1 标准字段

`AjaxResult` 继承 `HashMap<String, Object>`，核心字段如下：

- `code`：业务状态码
- `msg`：提示信息
- `data`：业务数据，可选

部分接口还会在返回中附加其他字段，例如：

- `token`
- `user`
- `roles`
- `permissions`
- `uuid`
- `img`
- `url`
- `fileName`

### 3.2 成功返回示例

无数据返回：

```json
{
  "code": 200,
  "msg": "操作成功"
}
```

带数据返回：

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {
    "userId": 1,
    "userName": "admin"
  }
}
```

### 3.3 常见业务返回

登录接口：

```json
{
  "code": 200,
  "msg": "操作成功",
  "token": "eyJhbGciOiJIUzI1NiJ9..."
}
```

验证码接口：

```json
{
  "code": 200,
  "msg": "操作成功",
  "captchaEnabled": true,
  "uuid": "f9d6c2f1...",
  "img": "Base64内容"
}
```

用户信息接口：

```json
{
  "code": 200,
  "msg": "操作成功",
  "user": {},
  "roles": [],
  "permissions": []
}
```

### 3.4 失败返回示例

```json
{
  "code": 500,
  "msg": "新增用户'admin'失败，登录账号已存在"
}
```

或：

```json
{
  "code": 403,
  "msg": "没有权限，请联系管理员授权"
}
```

## 4. `TableDataInfo` 分页结构

分页接口通常返回如下结构：

- `code`：状态码
- `msg`：提示信息
- `rows`：当前页数据列表
- `total`：总记录数

示例：

```json
{
  "code": 200,
  "msg": "查询成功",
  "rows": [
    {
      "userId": 1,
      "userName": "admin"
    }
  ],
  "total": 1
}
```

## 5. 分页请求参数

项目分页参数由公共分页组件读取，常用参数如下：

- `pageNum`：页码，默认 `1`
- `pageSize`：每页条数，默认 `10`
- `orderByColumn`：排序字段，前端通常传驼峰命名
- `isAsc`：排序方向，支持 `asc`、`desc`、`ascending`、`descending`
- `reasonable`：分页合理化，默认 `true`

请求示例：

```http
GET /system/user/list?pageNum=1&pageSize=10&orderByColumn=createTime&isAsc=descending
```

说明：

- 后端会将 `orderByColumn=createTime` 转换为下划线风格排序字段
- 大部分列表接口都遵循这个分页规则

## 6. 文件上传下载返回

### 6.1 单文件上传

`POST /common/upload`

成功返回：

```json
{
  "code": 200,
  "msg": "操作成功",
  "url": "http://localhost:9531/profile/upload/2026/04/xx.png",
  "fileName": "/profile/upload/2026/04/xx.png",
  "newFileName": "xx.png",
  "originalFilename": "logo.png"
}
```

### 6.2 多文件上传

`POST /common/uploads`

成功返回：

```json
{
  "code": 200,
  "msg": "操作成功",
  "urls": "url1,url2",
  "fileNames": "file1,file2",
  "newFileNames": "a.png,b.png",
  "originalFilenames": "a.png,b.png"
}
```

说明：

- 多文件上传结果使用英文逗号拼接，不是 JSON 数组
- 前端使用前通常需要自行 `split(',')`

### 6.3 通用下载

以下接口返回的是文件流，不是 `AjaxResult`：

- `GET /common/download`
- `GET /common/download/resource`

处理方式：

- 前端按二进制流下载处理
- 成功时通常通过响应头 `Content-Disposition` 获取文件名
- 失败时不一定返回标准 JSON

## 7. 导入导出接口特点

### 7.1 Excel 导出

例如：

- `POST /system/user/export`
- 应急模块中的 `export` 接口

这类接口通常直接写入 `HttpServletResponse` 输出流：

- 返回内容为 Excel 文件
- 不返回标准 JSON

### 7.2 Excel 导入

例如：

- `POST /system/user/importData`

通常返回：

```json
{
  "code": 200,
  "msg": "导入成功，共 10 条，更新 2 条"
}
```

### 7.3 异步导出接口

`epmis-process` 中还存在一组异步导出接口：

- `GET /export/excel`
- `GET /export/status/{taskId}`
- `GET /export/download/{taskId}`

其返回形式与常规接口不同：

- `/export/excel`：返回 `{"taskId":"xxx"}`
- `/export/status/{taskId}`：返回 `{"status":"RUNNING"}`
- `/export/download/{taskId}`：返回文件流

说明：

- 这一组接口使用 `ResponseEntity<String>` 或直接输出文件，不走 `AjaxResult`
- 前端不能按统一响应拦截器直接解析

## 8. 特殊返回接口

以下接口不完全遵循统一结构：

- `GET /getLoginUser`：直接返回 `LoginUser` 对象
- `GET /system/user/allUserInfo`：直接返回用户列表 `List<SysUser>`
- `GET /export/excel`：返回简单 JSON 字符串
- `GET /export/status/{taskId}`：返回简单 JSON 字符串
- 文件下载类接口：返回二进制流

建议：

- 对这些接口在前端单独封装
- 避免强制套用统一 `code/msg/data` 解析逻辑

## 9. 业务状态码说明

项目中定义的主要状态码如下：

- `200`：成功
- `201`：创建成功
- `202`：已接受
- `204`：成功但无返回体
- `400`：请求参数错误
- `401`：未认证或登录失效
- `403`：无权限
- `404`：资源不存在
- `405`：请求方法不支持
- `409`：资源冲突
- `415`：不支持的媒体类型
- `500`：系统异常或业务失败
- `601`：警告类返回

说明：

- 实际业务接口最常见的是 `200`、`401`、`403`、`500`
- `601` 由 `AjaxResult.warn(...)` 使用

## 10. 常见错误返回

### 10.1 未登录或 Token 无效

认证失败时，接口返回：

```json
{
  "code": 401,
  "msg": "请求访问：/xxx，认证失败，无法访问系统资源"
}
```

前端建议：

- 统一拦截 `401`
- 清理本地 token
- 跳转登录页

### 10.2 权限不足

权限校验失败时，接口返回：

```json
{
  "code": 403,
  "msg": "没有权限，请联系管理员授权"
}
```

前端建议：

- 统一提示无权限
- 不要自动重定向登录

### 10.3 参数校验失败

参数绑定、校验失败时，通常返回：

```json
{
  "code": 500,
  "msg": "字段校验失败提示"
}
```

说明：

- 虽然语义上更接近 `400`，但当前全局异常处理大多仍按 `AjaxResult.error(...)` 返回
- 前端不要只按 HTTP 状态码判断业务失败，也要读取 JSON 中的 `code`

### 10.4 请求方式错误

例如接口要求 `POST`，实际发了 `GET`，通常返回：

```json
{
  "code": 500,
  "msg": "Request method 'GET' is not supported"
}
```

## 11. 前端处理建议

- 先判断接口是否为文件流或特殊返回接口
- 普通接口优先按 `code` 判断成功失败，不要只依赖 HTTP 状态码
- 分页接口统一读取 `rows` 与 `total`
- 上传接口从 `url`、`fileName`、`originalFilename` 取值
- 多文件上传结果先拆分逗号字符串再组装数组
- 对 `401`、`403`、`500` 分别做不同的全局提示策略

## 12. 推荐封装方式

前端可按以下三类接口分别封装：

1. 普通 JSON 接口：按 `AjaxResult` 解析
2. 分页列表接口：按 `TableDataInfo` 解析
3. 文件流与特殊接口：关闭统一响应解析，单独处理
